SECTION 3: HIGH-RISK SIGNATURE GATE (高風險動作必簽字)
================================================================================

RULE 3.1 - Signature Required Actions:
The following actions REQUIRE explicit human authorization:

A. MERGE OPERATIONS
   - Any git merge into canonical branch
   - Any merge of candidate branches
   - Risk: Can introduce unauthorized changes

B. PUSH OPERATIONS
   - Any git push to remote
   - Any force-push operation
   - Risk: Can modify remote state unexpectedly

C. FORCE-PUSH OPERATIONS
   - git push --force or --force-with-lease
   - Risk: Can rewrite history, bypass validation
   - Special: --force-with-lease is fail-closed but still requires authorization

D. LANE RELEASE
   - Removing STOP_NOW.flag
   - Changing run_state from stopped to active
   - Risk: Can enable unauthorized automation

E. GOVERNANCE INFRASTRUCTURE CHANGES
   - Modifying .githooks/pre-push
   - Modifying validators (check_commit_message.py, etc.)
   - Modifying manifests/governance_round.yaml
   - Risk: Can bypass or weaken governance controls

RULE 3.2 - Signature Process:
1. System detects high-risk action
2. System PAUSES execution
3. System requests explicit authorization via:
   - Telegram approval keyboard (approve/reject/pause)
   - Or explicit human message confirmation
4. System logs authorization timestamp and content
5. Only after authorization: proceed with action
6. If rejected: abort and report failure

RULE 3.3 - No Auto-Approval:
- System MUST NOT auto-approve any high-risk action
- System MUST NOT use --no-verify to bypass hooks
- System MUST NOT modify hooks/validators to self-approve

================================================================================

RULE 5.1 - Status vs Content Consistency:

- System MUST detect contradiction
- System MUST pause or block
- Example: status=completed but blockers_found is not empty

RULE 5.2 - Field Completeness Check:
If required fields are missing from RETURN_TO_CHATGPT:
- System MUST detect missing fields

- System MUST NOT return incomplete reports

RULE 5.3 - Legal Conflict Detection:
If round actions conflict with governance law:
- System MUST detect conflict
- System MUST block immediately
- Example: Modifying master when law says "不得碰 master"
