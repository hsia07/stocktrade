## 十八、Law-0416 Phase 2B Usage Exhaustion Supplementary Rules

第 7 條【制度目的】本條為 Law-0416 Phase 2B 之補充規則，針對 usage exhaustion、Insufficient Balance、API quota 耗盡、model unavailable 等資源中斷情境，明確定義 pause 行為、禁止行為與 resume 條件，防止自動化系統在資源中斷後產生非預期行為。

第 8 條【資源中斷之分類】Phase 2B pause 原因應至少區分以下子分類：
- **insufficient_balance**：使用者帳戶餘額不足，導致 API 調用被拒
- **api_quota_exhausted**：API 配額（rate limit / token quota）已耗盡
- **model_unavailable**：模型端暫時不可用（503 / overload / deprecation）
- **usage_source_unavailable**：OpenCode 或其他自動化工具無法取得（usage_source_available = FALSE）
- **repeated_failure**：連續錯誤次數 ≥ 3（consecutive_failure_count）
- **silence_or_stall**：系統無回應 ≥ 30 分鐘（stuck_duration）

第 9 條【Usage Exhaustion ≠ Code Failure】資源中斷（insufficient_balance / api_quota_exhausted / model_unavailable / usage_source_unavailable）不屬於 code failure。不得：
- 視為程式錯誤而觸發 failure 重試邏輯
- 自動切換至不同模型規避配額限制
- 自動輪換 API key 繞過額度限制
- 視為該輪 candidate 可被跳過或自動推進下一輪

第 10 條【禁止自動續跑】在以下情境下，系統不得自動續跑：
- 充值後不得自動恢復執行
- 額度恢復後不得自動恢復執行
- 模型恢復可用後不得自動恢復執行
- API key 更換後不得自動恢復執行
- 唯一合法恢復方式：使用者手動 /start 或明確書面恢復授權

第 11 條【手動 /start Resume 義務】當因 usage exhaustion / insufficient balance / API quota / model unavailable 觸發 pause 後，僅以下方式可恢復：
- 使用者傳送 /start 指令（Telegram 或 Panel）
- 使用者以可追溯方式明確授權 resume
- 以上二者均須於 resume 前完成 pre-resume checks
- 任何自動化恢復機制均屬違法

第 12 條【Pre-Resume Checks】resume 前必須執行以下檢查，全部通過始可 resume：
1. local HEAD == remote HEAD（canonical 一致）
2. git status clean（無未授權 dirty 檔案）
3. run_state == stopped 或 paused（合法狀態）
4. main_control_loop 未啟動
5. R030 或任何輪次 dispatch 未啟動（除非已明確授權）
6. trading / broker / execution / live 全部 FALSE
7. order_execution_allowed == FALSE
8. 前一輪 RETURN_TO_CHATGPT 為有效正式主體輸出
9. pause 原因已確認解除（餘額充足 / 額度已恢復 / 模型可用）
10. 使用者已明確授權 resume

第 13 條【法律效果】違反第 9–12 條之自動行為（auto-retry / auto-model-switch / auto-key-switch / auto-continue / auto-resume），視為治理違規，該輪 candidate 標示為 blocked，不得推進。若已 merge 或 push，則須啟動 rollback 程序。

## 五、多輪候選預製模式法典

第 7 條【制度目的】為兼顧高風險工程專案之治理嚴格性與長時自動化效率，允許在明確授權下，將多個後續輪次之成果先行自動收斂為各自獨立之 candidate，以供使用者回來後集中審核；但此制度不得改變正式通過、正式 merge、正式收口之人工最終放行原則。

第 8 條【定義】多輪候選預製模式，指代理、排程器、OpenCode、Aider 或其他輔助工具，於使用者明確授權之輪次範圍內，得依序完成各輪最小必要施工、驗證、證據收集與 candidate 建立，但不得因此視為該輪已正式通過。

第 9 條【啟動前提】本模式非預設常態流程。僅於使用者以可追溯方式，明確授權起始輪次、終止輪次、適用 Phase、禁止事項與停止條件時，始得啟動。

第 10 條【法律效果限制】本模式僅放寬『可先預製候選』之流程，不放寬正式通過、正式 merge、正式收口、正式進下一 Phase 等控制權。不得因後一輪 candidate 已完成，而推定前一輪已正式收口。

第 11 條【單輪獨立義務】在多輪候選預製模式下，每一輪仍必須保有獨立身份，不得混帳。至少應具備獨立 round_id、candidate_id、candidate branch 或等價隔離機制、evidence package、RETURN_TO_CHATGPT、formal status code、candidate commit 與可回滾審計鏈。

第 12 條【禁止冒充正式通過】任何代理、排程器、OpenCode、驗收腳本或人工審核官，不得因某輪已進 candidate、已完成測試、已建立 evidence package，便表述為該輪已正式通過、可直接 merge 或可無條件進下一輪。

第 13 條【可前推範圍】本模式下，後續輪得在前一輪尚未正式 merge 前，繼續進行下一輪之 candidate 預製；但後續輪次之法律狀態僅得為 technical_unfinished、blocked、environment_unavailable、candidate_ready_awaiting_manual_review 或其他已明文定義且不構成正式通過之狀態碼。

第 14 條【停止規則】只要任一輪出現 law_undefined、授權外檔案被觸及、hard gate 無法合法通過、任務尺寸超過可合理拆分範圍、審計鏈不足以保證可追責、或命中使用者明訂停止條件，後續自動預製應立即停止，不得自行恢復。

第 15 條【集中審核權】使用者得於多輪候選預製完成後，一次性審核多個輪次之 candidate；但集中審核僅屬審核便利，不改變各輪獨立性，不得因此將多輪視為單一輪次通過。

第 16 條【禁止整包自動 merge】除非法典另有更高層正式授權，否則多輪候選預製模式下，不得將多輪 candidate 視為可整包自動 merge 之集合。每輪仍應逐輪進行候選核准、review / promote、merge 決策、merge 執行與收口留痕。

第 17 條【與不可跳輪法之關係】本模式構成對『前一輪未正式通過前不得進入下一輪』之有限例外，但僅限於 candidate 預製層級，不及於正式通過、正式 merge、正式收口或下一 Phase 啟動。

## 六、面板 Start 與自動授權法典

第 18 條【正式啟動前提】只有在 panel Start / Resume 與實際後端腳本、bridge endpoint、wrapper、排程狀態寫入之間存在明確、可驗證、可重跑之綁定時，Start 才可被視為正式啟動與授權行為。若 Start 僅為前端 UI reset、文字變更或 drain timer 清理，不得視為正式授權。

第 19 條【Start 的法律效果】若正式啟動前提成立，且使用者於當前法典與 baseline 中已明示該 Phase 可啟用多輪候選預製模式，則使用者自面板按下 Start，應自動理解為：授權該 Phase 後續所有尚未處理輪次，在合法停止條件未命中前，可自動依序預製到各自 candidate。

第 20 條【Start 不構成之授權】面板 Start 不當然構成下列授權：approve、promote、merge、push master、越權碰 forbidden path、放寬 hard gate、放寬證據規則、跳過單輪 RETURN_TO_CHATGPT、或跳過任一輪完整法條重檢。

第 21 條【停止點】即使 Start 已授權該 Phase 後續輪次自動預製，只要任一輪未達 candidate_ready_awaiting_manual_review、命中停止條件、進入 review / merge 決策點、或需要新的權限判定，流程即須停止並等待使用者。

第 22 條【優先解釋】若對面板 Start 是否構成某項更大授權有疑義，一律採最保守解釋：僅視為 candidate 預製授權，不視為 merge、push 或下一 Phase 啟動授權。

## 七、候選門檻與完整功能判定法典

第 23 條【候選不是只看摘要】OpenCode 在判斷某輪是否可提交至 candidate 時，不得只看最後修改片段、commit message、單段摘要、單張 diff、單一測試或 AI 口頭回報；必須回到該輪完整法條，逐項檢查該輪要求的所有功能、所有必要契約、所有必要情境與所有直接 FAIL 條件。

第 24 條【缺一不可】凡屬該輪法條所明列之必要功能、必要 API / schema / state contract、必要成功情境、必要失敗情境、必要邊界情境、必要恢復情境、必要非法情境、必要證據與必要驗收腳本輸出，只要有一項未完成、未可運行、未驗證、未留痕，該輪即不得標記 candidate_ready。

第 25 條【功能完整性義務】候選門檻必須同時滿足：一、該輪唯一主題所有必做功能皆已具備；二、功能不僅存在，且可正常運行；三、必要契約與狀態一致；四、必要測試與驗證通過；五、evidence package 完整；六、hard gate 真正執行且 PASS；七、無 fabricated evidence；八、無越權修改。

第 26 條【整輪重檢義務】OpenCode 判定 candidate 時，必須以整輪法條為唯一依據，重新審核完整輪次，不得只看本輪最後一次施工的檔案集合。任何遺漏即視為未達候選門檻。

第 27 條【候選法定最低證據】每輪 candidate 至少應具備 task.txt、aider.log、candidate.diff、report.json、該輪可重跑測試結果、必要 state / API 證據、必要 event log、必要驗收文件與完整 RETURN_TO_CHATGPT 主體輸出。缺一不可。

第 28 條【形式與實質雙成立】就算 evidence package 齊全，若功能實質上未完整具備；或功能看似存在但必要證據不足；二者任一不足，皆不得進 candidate。

## 八、merge 前完整重審法典

第 29 條【merge 前不得只看候選摘要】當候選已進入 candidate_ready_awaiting_manual_review 或 review branch 階段，任何代理、OpenCode、ChatGPT 或高階審核者，在判斷是否可 merge 前，不得僅依 RETURN_TO_CHATGPT 摘要、candidate 名稱或單段 acceptance 摘要做決策。

第 30 條【merge 前完整重審義務】於判斷候選是否可以 merge 時，必須重新讀取 repo 內與該輪有關之實際檔案、測試、契約、驗收記錄、run record、evidence package 與該輪完整法條，重新審核一遍該輪是否完整具備所有要求功能且能正常運行。

第 31 條【重審範圍】merge 前重審至少應涵蓋：一、該輪唯一主題；二、所有必做功能；三、所有必要契約；四、所有必要情境與直接 FAIL 條件；五、實際 repo 檔案與測試；六、candidate / review branch 差異；七、hard gate 是否真實通過；八、是否仍存在未解決風險。

第 32 條【重審結論效力】若 merge 前重審發現該輪仍有功能缺漏、契約不一致、測試不穩定、證據不足、或任何直接 FAIL 條件，則即使該候選先前曾被標為 candidate_ready，也不得 merge。

第 33 條【高階審核官義務】ChatGPT 在給出 merge 建議或 merge 指令前，應明示其係基於完整重審，而非僅基於 OpenCode 摘要；若未完成完整重審，不得提供 merge 指令。

## 九、merge 前人工簽字義務法典

第 34 條【merge 前明示授權義務】當流程已進入 review branch，且客觀上已達到可由人工決定是否 merge 之決策點時，任何代理、排程器、OpenCode、ChatGPT 或其他輔助系統，不得預先提供 merge 指令，不得逕自推進 merge 流程，亦不得以預設同意、默示同意或流程慣例視為已獲授權。

第 35 條【簽字的法律效果】僅於使用者以可追溯方式明確表示『好』『同意』『簽字同意』或其他等價明示授權後，始得提供 merge 指令、執行 merge、或進入任何足以發生等價法律效果之後續流程。

第 36 條【不得以沉默替代】沉默、未反對、先前慣例、模糊語句、閱讀未回覆、或長時間未處理，均不得視為 merge 授權。

第 37 條【違反效果】若未先取得使用者明示簽字，即先行提供 merge 指令、建議直接合流、或執行 merge，視為越權推進正式收口；除非使用者事後追認，否則程序上應視為違規。

## 十、RETURN_TO_CHATGPT 正式主體輸出法典

第 38 條【正式主體輸出義務】RETURN_TO_CHATGPT 不是摘要區，而是正式主體輸出區。凡足以影響輪次判定、candidate 狀態、approve / promote / merge、blocked、technical_unfinished、candidate_ready、修法成立與否或下一步流程之資訊，均必須完整寫入 RETURN_TO_CHATGPT，不得僅存在於前文說明、人類可讀摘要或其他非正式區塊。

第 39 條【前文僅供輔助】正式輸出之前文 1–5 點、分段說明、表格、敘述性總結或其他人類可讀內容，僅得作為輔助閱讀材料，不得比 RETURN_TO_CHATGPT 更完整，不得承載正式主體結論，不得形成『前文較完整、正式區僅寫摘要』之輸出結構。

第 40 條【最低必備欄位】RETURN_TO_CHATGPT 至少應載明：round_id、task_type、reply_id、status 或 formal_status_code、candidate_id / branch / commit（若有）、files_modified 或 NONE、證據 / 驗證 / 測試結果、escalation_required、next_action，以及其他足以影響是否放行、是否退回、是否 blocked 之關鍵結論。

第 41 條【不一致之效果】若前文內容與 RETURN_TO_CHATGPT 不一致，或關鍵資訊僅存在於前文而未進入 RETURN_TO_CHATGPT，視為正式輸出無效，整份回報應退回重做，不得作為驗收、排程、候選核准、promotion、人工審核或輪次推進之依據。

第 42 條【唯一正式依據】任何代理、排程器、人工審核官、驗收腳本或面板，於正式判讀時，應以完整且有效之 RETURN_TO_CHATGPT 為唯一正式主體輸出依據；前文僅供輔助閱讀，不得取代之。

第 43 條【回報交付協議——聊天室同步輸出義務】
一、當使用者明確要求「回報給 ChatGPT」或等價指令時，OpenCode 必須同步執行以下兩項交付：
　(1) 將完整回報內容寫入 latest_return_to_chatgpt.runtime.txt；
　(2) 將相同完整回報內容輸出至當前聊天室對話中，供使用者直接檢視。

二、僅在「自動模式」經使用者明確啟用時，OpenCode 得豁免聊天室輸出義務，僅執行檔案寫入。自動模式之啟用必須以可追溯方式記錄於 state.runtime.json 之 desired_action 或等價欄位。

三、禁止僅寫入檔案而不在聊天室呈現（自動模式除外）。違反本條視為回報交付不完整，該回報不得作為驗收、排程、候選核准、promotion、人工審核或輪次推進之依據。

四、本條為強制執行規範，不得因方便、效能、或「檔案已寫入即可」之慣例而省略聊天室同步輸出。

## 十一、OpenCode / Aider / 面板執行補充法典

第 43 條【OpenCode 角色】OpenCode 之正式角色為橋接官、任務格式官、回報官。其職責限於承接當輪唯一任務或已合法授權之多輪候選預製任務、在授權範圍內施工或派工、執行被允許之驗證、與輸出可審計結果。

第 44 條【OpenCode 禁止事項】OpenCode 不是自由施工員、不是環境修復員、不是最終放行者。除非法典明確授權，不得自行安裝套件、修改 PATH 或永久環境變數、清理 working tree、push、merge、宣布正式通過或擴權碰 forbidden path。

第 45 條【Aider 負荷上限】凡透過 Aider 施工之任務，原則上一輪只能有一個明確小任務，且應優先只授權一個檔案。若確有必要，最多只允許兩個高度相關之中小型檔案同輪送入。不得把超大單檔、核心交易主流程大檔、需要整 repo 理解之任務、或規劃 / 環境修復 / 改檔 / 驗證混成一次複合任務直接丟給 Aider。

第 46 條【Aider 最小上下文義務】每次呼叫 Aider 前，應優先以 --file、--message、--no-auto-commits、--no-dirty-commits 組成最小上下文；若仍超出負荷，應先拆任務而非強送。

第 47 條【證據守門】validate_evidence.ps1 或其後續正式替代腳本，屬 candidate 硬門之一部分。未真正執行且通過時，不得以人工摘要、手動比對或口頭保證取代。

第 48 條【Panel / wrapper 啟動鏈】所有桌面捷徑、bat/cmd/ps1 wrapper、bridge endpoint、panel 按鈕、-File 路徑與引號拼接，均屬正式啟動鏈的一部分。若存在空路徑、錯誤引號、腳本不存在或僅前端 reset，即屬啟動器故障，不得誤判為核心工具鏈可用。

## 十二、Phase 級主核心 Merge 治理法典

第 49 條【Phase 內單輪收口與 Phase 級收口之區別】Phase 內各輪得依正式流程進入 candidate、review、merge 到 work branch、push 到 origin/work branch；單輪進入 work branch，僅表示該輪在 Phase 工作主線內已完成收口，不等於該輪可直接進入 master / 主核心分支。

第 50 條【Phase 級總審核義務】Phase 內所有預定納入輪次皆完成單輪收口到 work branch 後，必須先執行 Phase 級總審核；未完成 Phase 級總審核前，任何輪次或整個 Phase 不得 merge 進 master / 主核心分支。

第 51 條【Phase 級主核心 Merge 之獨立性】Phase 級主核心 merge 屬獨立決策點，與單輪 merge 到 work branch 分屬不同授權層級。單輪已 merge 到 work branch，不得推定為該輪或該 Phase 可直接進入主核心。

第 52 條【禁止自動整包 Merge】OpenCode、排程器、代理或任何自動化流程，不得因 Phase 內多輪皆已進入 work branch，便自動或整包將該等輪次 merge 進 master / 主核心分支。

第 53 條【未完成輪次之阻塞效果】Phase 內若仍存在 candidate only、review only、technical_unfinished、blocked 或其他非正式收口狀態之輪次，原則上不得進行該 Phase 之主核心 merge；除非使用者明示授權例外處理並記錄理由。

第 54 條【使用者明示簽字義務】OpenCode 到達 Phase 級主核心 merge 決策點時，必須先停止流程，等待使用者以可追溯方式明示簽字同意。未獲使用者明示簽字前，OpenCode 不得提供主核心 merge 指令、不得執行 master merge、不得建議直接合流。

第 55 條【不得推定同意】不得因使用者曾簽字同意單輪 merge 到 work branch、或曾授權多輪候選預製、或長時間未處理，便推定使用者已同意 Phase 級主核心 merge。

第 56 條【work branch 定位】work branch（如 work/r006-governance）為 Phase 內收口工作主線，供多輪逐步匯集與 Phase 內整合測試之用；master / 主核心分支為正式發布與生產就緒之獨立保護層。

第 57 條【主核心 Merge 前檢查清單】判定是否可進行 Phase 級主核心 merge 前，必須確認：
一、該 Phase 預定納入輪次是否皆已收口到 work branch；
二、是否仍存在 candidate only、review only、technical_unfinished、blocked 狀態之輪次；
三、是否已完成 Phase 級總審核，確認各輪功能協同正常、無互相衝突、無回歸失效；
四、是否已取得使用者明示簽字同意。

第 58 條【違反效果】違反本章規則、未經 Phase 級總審核或使用者明示簽字即執行主核心 merge，視為越權推進正式收口，應立即停止並回報 blocked；已發生之 merge 應視為待覆審狀態，等待使用者追認或回滾。

## 十三、Shell / Git 非互動執行治理法典

第 59 條【非互動執行義務】OpenCode 執行 git、shell、merge、push、switch、verify 或其他命令列流程時，必須採非互動模式，不得打開 editor、pager、或等待任何人工輸入。

第 60 條【merge 命令非互動參數】凡執行 git merge，必須使用可避免互動之參數，例如 --no-edit、--no-ff、或等價非互動設定；不得以預設模式執行，導致可能進入互動編輯器等待人工輸入。

第 61 條【前置最小檢查義務】執行 switch、merge、push、verify 前，必須先做最小必要前置檢查：一、確認分支存在（git rev-parse --verify）；二、確認工作目錄狀態（git status --porcelain）；三、確認授權範圍內操作對象。未做前置檢查即執行高風險操作，視為程序瑕疵。

第 62 條【超時中止義務】shell 或 git 步驟若超過固定上限時間（建議 15 秒）仍未完成，必須立即停止並回報：一、technical_unfinished，或二、blocked。不得停在「思考中」、「處理中」或等價無限掛起狀態。

第 63 條【merge conflict 立即 abort】若 merge 遭遇 conflict，必須立即執行 git merge --abort，且不得自行進入交互式修復、不得嘗試自動解決衝突、不得詢問使用者如何處理。abort 後應回報 technical_unfinished，等待進一步指示。

第 64 條【失敗完整回報義務】若 shell 或 git 步驟失敗，必須完整回報：一、executed_command（實際執行之命令）；二、stdout（標準輸出）；三、stderr（錯誤輸出）；四、failure_point（失敗點描述）。不得只回報「失敗」或「出錯」而不提供具體證據。

第 65 條【禁止多輪猜測式嘗試】不得因前一次命令失敗，便自行進行多輪猜測式 shell 嘗試、變更參數重試、或切換替代方案。每次失敗應停止並回報，等待明確授權後方可進行下一步。

第 66 條【非互動規則之硬性】本章規則屬治理硬限制，適用於所有 OpenCode 執行之 git、shell、驗證、merge、push、switch、verify 任務。不得因方便、經驗、或「通常不會有問題」而省略非互動參數或超時保護。

## 十四、Governance Drift 治理與單輪分支管理規則

第 67 條【單輪分支禁改治理檔】單輪 candidate / review branch 原則上不得修改治理檔。所謂治理檔，至少包含：
一、CURRENT_GOVERNANCE_BASELINE.md
二、_governance/law/00_入口與使用規則/ 目錄下所有檔案
三、_governance/law/04_交易系統法典補強版_20260416_修正版.md
四、_governance/law/readable/ 目錄下所有檔案

第 68 條【治理檔修改屬性】治理檔修改屬治理任務，非單輪功能任務。單輪功能實作（如 R-007 silence detection、R-008 state machine、R-009 command priority）不得夾帶治理檔變更。

第 69 條【單輪分支開出基準】新的單輪 branch 必須從最新 work branch（如 work/r006-governance）開出；不得從落後於 work branch 的基底開出，以免帶入舊版治理文件。

第 70 條【Governance Drift 檢查義務】單輪進 approve / promote / merge 前，必須先做 governance drift 檢查。檢查方式：比較 target branch 與最新 work branch 之間的治理檔差異。

第 71 條【Governance Drift 判定標準】若 git diff --name-only <work_branch>...<target_branch> 命中第 67 條所列之治理檔集合，即判定 governance drift = true。

第 72 條【Drift 發現時之處理】發現 governance drift 時，必須 blocked，不得繼續 approve / promote / merge。OpenCode 必須立即停止並回報：
一、target branch 名稱
二、base work branch 名稱
三、drifted governance files 清單
四、建議下一步（rebase onto latest work / replay branch / split governance changes into separate task）

第 73 條【Rebase / Replay 義務】若 branch 基底落後 work branch，應先 rebase / replay / 重建，不得硬 merge。Rebase 後必須重新執行 governance drift check，直至 drift = false。

第 74 條【單輪 merge 內容限制】單輪 merge 優先只帶入該輪功能內容與該輪 evidence；治理檔不得被舊 branch 汙染回流。若 merge 後發現 governance file 被覆蓋為舊版本，視為 merge 失敗，應立即回滾或重建。

第 75 條【Governance Drift Check 之硬性】本章規則屬治理硬限制，適用於所有單輪 approve / promote / merge 流程。必須以可重跑之 hook 或驗證腳本落地，不得只憑人工記憶或口頭提醒。

### 第一節　Backtest / Replay 正式驗收規則

第 83 條【Backtest 工具角色限制】TradingView / Pine Script / DeepTest 或其他第三方圖表回測工具，僅得作為 L0 草稿、視覺輔助、初步 sanity check 使用。
正式驗收不得以 TradingView-only 或任何第三方工具獨佔分析作為通過依據。

第 84 條【正式驗收強制要求】任何策略、信號、AI 決策、風險閘門、執行邏輯、paper-live 過渡、損益計算、市場現實層、或交易邏輯相關輪次，其正式驗收必須同時滿足以下條件，缺一即 FAIL：
一、Python backtest compatible：核心回測邏輯必須以 Python 實作且可重跑，不得依賴第三方封閉平台。
二、Event replay compatibility：必須支援事件驅動回放，可逐事件重演決策順序、成交結果與狀態變遷。
三、Decision trace replay：必須支援決策軌跡回放，可重建每筆決策之輸入、信號、信心、風險閘門結果與最終下單與否。
四、Order / Fill / PnL schema：必須有正式 order schema、fill schema、pnl schema，回測輸出必須符合該 schema 且可機器驗證。
五、Risk Gate veto replay：必須支援風險閘門否決回放，可重演每筆 risk gate 是否 veto、veto 理由、veto 後最終動作。
六、Net-of-cost metrics：所有績效指標必須扣除完整成本（手續費、交易稅、滑價、借券費、資金成本），不得使用 gross metrics 冒充通過標準。

第 85 條【Replay / Backtest 證據要求】每輪若涉及回測或回放，evidence.json 至少應包含：
- replay_version：回放引擎版本
- sample_period：樣本期間（起訖日期）

- python_backtest_compatible：true/false
- event_replay_compatible：true/false
- decision_trace_replay_compatible：true/false
- order_fill_pnl_schema_validated：true/false
- risk_gate_veto_replay_compatible：true/false

### 第二節　Market Reality Layer

第 86 條【市場現實驗證義務】交易系統不得只驗證方向是否正確。所有涉及下單決策、執行模擬、paper-live 過渡之輪次，必須同時驗證市場現實層面，包含但不限於：

一、成本模型：券商手續費（含買賣方向差異、最低手續費）、交易稅（證交稅、期交稅）、零股/整股成本差異、借券費、融資利息。
二、滑價模型：掛單深度、買賣價差、波動率影響、開收盤集合競價、漲跌停距離、事件衝擊、下單量占市場日均成交量比例。
三、流動性 veto：成交量不足、委買委賣深度不足、成交機率不足、接近漲跌停時不得硬追。
四、Fill realism：partial fill（部分成交）、unfilled（未成交）、rejected（拒單）、timeout cancel（逾時取消）、cancel-fill race（取消與成交競爭）、delayed broker report（延遲回報）、duplicate fill report（重複成交回報）之情境處理。
五、Broker/local reconciliation：券商回報與本地狀態必須最終一致，不一致時以 broker 回報為準並觸發 reconciliation 流程。
六、Expected vs actual slippage drift：預期滑價與實際滑價持續偏差超過 threshold 時，必須 recalibrate 或 downgrade。
七、Expected vs actual fill-rate drift：預期成交率與實際成交率持續偏差超過 threshold 時，必須 recalibrate 或 downgrade。

第 87 條【不可成交時之行為】方向正確但成交不可行或不划算時（滑價過大、流動性不足、成本過高、風險超過 threshold），必須採取以下動作之一，不得無視市場現實強行下單：
- no-trade：放棄該筆交易
- reduce-size：縮小部位至可執行規模
- wait-confirmation：等待更佳流動性或價格改善後再評估
- abort：取消該訊號之整批交易

### 第三節　台股制度限制

第 88 條【台股強制限制】台股交易制度與美股 / crypto 有本質差異，不得套用 24/7 交易、無漲跌停、isolated margin、高槓桿、永遠可成交模型作為台股預設。所有交易邏輯、模擬、回測、執行、風控必須至少納入以下限制：

一、漲跌停 ±10%（當日最大漲跌幅限制），觸及漲跌停時不得以下單簿外價格強行交易。
二、T+2 交割與 cash availability：成交後第二營業日交割，可用買力須扣除未交割款項。
三、開盤集合競價（8:30-9:00 試算，9:00 開盤）與收盤集合競價（13:25-13:30 試算，13:30 收盤），集合競價期間不得以限價單指定於競價範圍外之價格。
四、盤中交易時段（9:00-13:25 連續競價），盤後零股交易時段（13:40-14:30 僅限零股）。
五、零股（1-999 股）與整股（1000 股為一張）差異：零股流動性較差、買賣價差較大、手續費最低可能不同。
六、停牌 / 暫停交易：公司召開股東會、重大訊息公布前、處置措施期間，該股票暫停交易或限制交易方式。
七、注意股 / 處置股限制：符合注意標準之股票可能被限制分盤交易、全額交割、或禁止當沖。
八、不得使用 isolated margin / portfolio margin / cross-margin 等不適用於台股之保證金模型作為下單放行依據。

### 第四節　Trading-Specific L0-L9 驗證梯

第 89 條【交易專用 L0-L9 驗證梯定義】以下為交易策略與信號專用之正式驗證梯，與程式碼流程驗證（L0 syntax → L9 production release）為不同概念，兩者均須通過方可升級 live：

- L0：Backtest analytic correctness（回測分析正確性）。策略邏輯在歷史資料上可重現、淨損益方向正確、基礎統計指標合理。
- L1：Walk-Forward（滾窗驗證）。策略在滾動視窗上表現穩定，非單一時期過適。
- L2：Monte Carlo / permutation test（蒙地卡羅 / 置換檢定）。隨機化檢定確認績效非隨機巧合。
- L3：Block Bootstrap（區塊拔靴法）。保留時間序列依賴結構之隨機重抽，檢定策略穩健性。
- L4：Bonferroni / multiple testing correction（多重檢定校正）。多策略、多參數、多標的同時測試時校正顯著性閾值。
- L5：Walk-Forward OOS（滾窗樣本外驗證）。滾窗留存樣本外期間之獨立驗證。
- L6：Regime segmentation（市場體制分割）。在不同市場體制（趨勢/盤整/高波動/低波動/事件期）下分別驗證策略表現。
- L7：Combinatorial Purged Cross-Validation（組合式純化交叉驗證）。避免資料洩漏之交叉驗證，訓練集與測試集之間設有 purge 視窗。
- L8：Probabilistic Sharpe Ratio / Reality Check / SPA（機率化夏普比率 / 真實性檢定 / 逐步 superior predictive ability）。統計檢定策略績效是否顯著優於基準。
- L9：Paper trading 6–12 months + live 前深度壓力測試。正式實盤前至少 6–12 個月模擬交易，含極端情境壓力測試、broker 回報對帳、成本衝擊分析。

第 90 條【L0-L9 驗證梯之法律效果】
一、回測本身僅為 L0，不得作為上線依據。
二、未通過 L4（多重檢定校正）與 L8（機率化顯著性檢定）之策略，不得進入 paper trading。
三、未通過 L7（組合式純化交叉驗證）與 L9（模擬交易 + 壓力測試）之策略，不得升級 live。
四、L0-L8 可在 research / sim / shadow 模式中完成；L9 必須在 paper trading 模式中執行，且 duration 不得少於 6 個月（或經使用者以可追溯方式明示縮短）。
五、formal acceptance 記錄必須包含：驗證版本（ladder_version）、樣本期間（sample_period）、資料版本（data_version）、設定版本（config_version）、net-of-cost metrics（扣除完整成本後之績效指標）。

### 第五節　Strategy Lifecycle Governance

第 91 條【策略生命週期狀態】每項交易策略至少應有以下生命週期狀態，且狀態轉移必須留痕：
- research_only：預設狀態。新策略產生時 default 為 research_only，不得直接影響實盤。
- shadow：僅觀察，不產生訂單，用於比對與累積信心。
- paper：模擬交易，使用模擬資金與真實市場條件。
- live_small：小資金實盤，有單筆與總曝險上限。
- live_normal：正常實盤，接受完整曝險。
- quarantined / disabled：自動降級或人工停用，不影響其他策略。

第 92 條【策略 Promotion 原則】
一、六大 AI / local arbitrator 可產生 strategy candidate，可自動建立 research_only 或 shadow 策略，但不得直接提高實盤權限。
二、Auto-promotion 只允許到 shadow / paper 層級，且需通過對應之正式 gate 驗證。
三、paper → live_small、live_small → live_normal、live 加資、live core logic change（策略本質邏輯變更）必須 human approval，不得 auto-promote。
四、human approval 必須以可追溯方式記錄，不得以沉默、未反對、或先前慣例視為已授權。

第 93 條【健康檢查與到期機制】
一、每項策略必須有 rolling health check（至少每週一次），檢查項目至少包含：pnl trend、drawdown、exposure、regime validity、signal decay 指標。
二、每項策略必須有 TTL（time-to-live），到期未 revalidation 者自動降級到 shadow 或 quarantined。
三、Regime validity：策略若在當前市場體制中表現低於 threshold，應自動 reduce-size 或 disable。
四、Decay detection：策略若出現持續績效衰退（consecutive loss、sharpe 持續下降、signal degradation），應自動降級或 quarantined。
五、Promotion expiry：paper 結果若超過 TTL（建議 90 天），不得作為 live approval 依據，必須重新驗證。
六、Revalidation before approval：任何 live 權限提升前，必須重新執行完整驗證梯（至少 L4-L9），不得沿用過期驗證結果。

第 94 條【自動降級原則】允許自動 downgrade / reduce-size / disable / quarantine，但絕對不允許自動提高實盤權限。自動降級無需 human approval，但必須留痕通知。

### 第六節　Deterministic Position Sizing / Portfolio Allocation

第 95 條【部位規模計算原則】
一、qty / allocation 必須由 deterministic math / risk engine 決定，不得由 LLM 決定。
二、每筆交易必須保留完整 allocation trace：strategy_id、strategy_version、capital_bucket、risk_budget、qty_formula、exposure_caps。
三、部位規模計算公式或等價規則：
   qty = min(trade_budget / risk_unit, ADV_or_liquidity_cap, portfolio_exposure_cap, strategy_cap, regime_cap, margin_or_cash_cap, drawdown_or_kill_switch_cap)
   其中各 cap 必須有具體數值與資料來源，不得使用 placeholder。
四、LLM 不得直接控制 qty；LLM 可提供建議，但最終 qty 必須由 risk engine 套用公式計算。

第 96 條【曝險上限集合】至少必須包含以下曝險上限，任一觸及即不得放大部位：
一、Total portfolio max exposure：組合總曝險不得超過絕對上限。
二、Per-symbol cap：單一標的曝險上限（金額或佔組合比例）。
三、Per-strategy cap：單一策略曝險總和上限。
四、Per-sector / per-theme cap：單一產業或主題曝險上限。
五、Correlation / crowding limit：相關性過高或過度集中之組合限制。
六、Daily loss limit：單日最大可接受損失，觸及後當日停止新交易。
七、Monthly loss limit：單月最大可接受損失，觸及後 reduce-size 或 pause。
八、Consecutive-loss cooldown：連續虧損達 threshold 後強制冷卻，不得下新單。
九、Cash reserve：最低現金儲備要求，不得用於交易。
十、T+2 cash availability：扣除 T+2 未交割款項後之實際可用資金。

第 97 條【Strategy 不得自行加資】策略本身不得自行提高其 capital bucket 或 risk budget。加資必須經過 human approval。Risk gate 必須可 veto 任何 allocation 變更。

### 第七節　NEA 淨期望優勢引擎

第 98 條【交易放行條件】交易放行必須同時滿足以下條件，任一不足即 no-trade：
一、net_edge > 0（淨期望優勢為正）。
二、fill_prob >= threshold（成交機率不低於最低門檻）。
三、expected_net_rr >= threshold（淨期望報酬風險比不低於門檻）。
四、regime_uncertainty <= threshold（市場體制不確定性不高於門檻）。
五、no veto（風險閘門、流動性閘門、制度閘門均未否決）。
六、market reality pass（市場現實層驗證通過）。
七、risk final gate pass（最終風險閘門通過）。

第 99 條【NEA 公式】淨期望優勢之正式計算公式：
   net_edge = p_hat * W_hat - (1 - p_hat) * L_hat - C - S - B - T - R - U

其中：
- p_hat：校準後之上漲/成功機率（必須是 calibrated probability，不得使用 raw confidence）
- W_hat：預期獲利金額（扣除成本前）
- L_hat：預期損失金額（扣除成本前）
- C：交易直接成本（手續費 + 交易稅）
- S：預期滑價成本
- B：借券/融資/保證金/機會成本
- T：延遲/排隊/資料時效成本
- R：尾部風險成本（extreme event premium）
- U：模型/體制/事件不確定性折價

第 100 條【NEA 約束】單一新聞、單一 K 線、單一技術指標、單一 AI 高分、單一社群訊號不得直接形成 buy/sell 決策，只能形成 candidate 進入正式評估流程。raw confidence 不得直接代入 p_hat；p_hat 必須經過校準。

### 第八節　Confidence Calibration

第 101 條【校準義務】
一、raw confidence 不得直接放行交易或放大倉位。
二、各 signal family / AI module 之 confidence 必須經過歷史校準，方可作為交易放行依據。
三、raw confidence 僅可在 dashboard / monitoring 中顯示參考值，不得出現在 risk gate / order / execution 決策鏈中。

第 102 條【校準記錄最低要求】每項校準必須保留以下記錄，可供 replay 與 audit：
一、calibration curve（校準曲線，預測機率 vs 實際發生率）。
二、Brier score（校準誤差分數）。
三、calibration slope / intercept（校準迴歸斜率與截距）。
四、calibration version（校準版本號）。
五、sample period（校準樣本期間）。
六、config version（校準設定版本）。

### 第九節　Setup SOP / Technical Signal Governance

第 103 條【Setup 強制結構】每個交易 setup 至少必須包含以下欄位，缺一不可：
一、market_context：當前市場狀態摘要（trend / regime / volatility / event）。
二、setup_type：setup 類型（trend_following / mean_reversion / breakout / event_driven / etc.）。
三、prerequisite_conditions：觸發前必須成立之前置條件。
四、entry_trigger：進場觸發條件（具體價格、指標值、事件等）。
五、confirmation_path：確認路徑（至少一個獨立確認信號，避免假突破 / 假訊號）。
六、invalidation / no-trade rules：失效條件與不交易規則。
七、stop / take-profit / RR math：停損、停利、預期報酬風險比計算。
八、liquidity / fill feasibility：流動性檢查與成交可行性評估。
九、source / data availability：信號來源、資料可得性與新鮮度。
十、traceable reason codes：可追溯之 reason code，供 replay 與事後審計。

第 104 條【Setup 使用限制】
一、A/B/C score 僅得作為 label 或參考分級，不得直接繞過 Risk Gate 或直接加倉。
二、NO TRADE 規則是 hard veto，不得因 A/B/C score 高分而 override。
三、FRVP / ICT / Footprint / DOM / Delta / Volume Profile 等進階概念，只有在資料可得且可 replay 時方可使用；資料不可得時必須標示 data_missing 並 no-trade，不得使用推估值或合成資料冒充真實資料。
四、每個 setup 之 validity 必須可用 Python / event-driven replay 驗證，不得 TradingView-only。

### 第十節　Event Source Credibility / Six-AI / LLM Governance

第 105 條【事件來源範圍】事件分析層至少應覆蓋以下來源：
一、新聞（國內外重大財經新聞）。
二、官方公告（主管機關、交易所、央行、財政部）。
三、三大法人買賣超與持倉變化。
四、籌碼面（集保戶股權分散、大額交易人）。
五、融資融券 / 借券餘額與變化。
六、期貨選擇權（未平倉量、put/call ratio、選擇權隱含波動率）。
七、重大政策（稅務、產業政策、外資管制）。
八、高影響社群（經篩選之可信賴財經社群、官方社群帳號）。

第 106 條【來源可信度分層】事件來源可信度至少分為以下層級，可信度較低之來源不得直接作為交易決策依據：
一、第一層（最高）：官方公告（交易所、主管機關）、券商回報（broker confirmation）。
二、第二層：三大法人數據、交易所公開統計數據。
三、第三層：經認證之財經新聞通訊社。
四、第四層：一般新聞媒體、研究報告。
五、第五層（最低）：社群媒體、論壇、未驗證消息。

第 107 條【事件層治理】事件層必須至少實作以下機制：
一、來源可信度評分：每筆事件帶有來源可信度分數。
二、去重保護：同一事件來自多來源時只處理一次。
三、舊聞污染保護：事件必須有 publish timestamp，超過 TTL 者不得進決策鏈。
四、事件壽命管理：每筆事件有有效期間，過期自動失效。
五、事件重要度分層：依影響範圍與程度分級，高影響事件需確認後方可反應。
六、衝突整合：多來源衝突時依可信度權重整合，無法整合時標示 conflicted 並 no-trade。

第 108 條【高影響未確認事件處理】高影響但未經官方確認之事件（rumor / unverified news / social media panic），只能採取以下動作之一，不得直接進場：
- watch：觀察後續發展，不動作。
- no-trade：明確放棄該事件相關交易。
- reduce-size：降低現有相關部位。
- wait-confirmation：等待官方或可靠來源確認後再評估。
- tighten-risk：收緊風控參數（降低曝險上限、提高 veto threshold）。
採取以上動作必須在 decision trace 中記錄事件 ID、可信度、採取動作、理由。

第 109 條【六大 AI 治理】六大 AI 必須是本地結構化決策模組 + 本地仲裁器，不得依賴外部 API 作為核心決策元件：
一、LLM/API 只能做摘要、解釋、研究、新聞解讀、複盤輔助。
二、交易放行 / risk veto / no-trade / final gate 不得依賴 LLM API。
三、LLM timeout / API fail / 額度用完 / 格式錯誤不得降低風控強度；若 LLM API 不可用，系統必須以最保守風控模式繼續運行，不得放寬交易限制補償資訊落差。
四、六 AI 仲裁輸入最低要求：每 AI 輸出立場、信心分數、主要理由、關鍵風險、是否 veto。

### 第十一節　R030 Applicability

第 110 條【R030 為必用輪次】R030「決策前檢查清單」為上述交易驗證與市場現實治理約束之必用輪次。R030 在施工時必須明確引用本章各節規則作為檢查清單項目，不得遺漏以下檢查點：
一、Market Reality pass（成本 / 滑價 / 流動性 / fill 模型驗證通過）。
二、NEA gate pass（net_edge > 0 且所有 release conditions 滿足）。
三、Risk Gate pass（所有風險閘門未 veto）。
四、Position sizing pass（qty 由 deterministic engine 計算且所有 cap 未觸及）。
五、Confidence calibration pass（所有使用之 confidence 已校準）。
六、Replay compatibility（決策鏈可完全回放）。
七、Taiwan constraints（台股制度限制已遵守）。
八、Single signal no direct trade（無單一訊號直接 buy/sell）。
九、Setup SOP（若使用 setup，必須有完整欄位且可 replay 驗證）。
十、Event credibility（事件來源含有可信度分層，高影響未確認事件未直接進場）。
十一、LLM not trade core（LLM 未用於交易放行 / risk veto / final gate）。

第 111 條【R030 違反效果】R030 若在施工或驗收時未引用本章規則，直接 FAIL。R030 仍為獨立輪次，不得在本 candidate 中 dispatch；本條僅規定 R030 必須參考本章規則。
