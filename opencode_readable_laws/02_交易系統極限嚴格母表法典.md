交易系統極限嚴格母表法典

法條級、驗收級、不可模糊解釋版

| 文件定位：這份文件不是白話導覽，而是未來所有審核、排程、手動檢查、opencode 指令、驗收腳本的『法律』。 使用原則：本法典優先於臨時對話、臨時猜測與口頭放寬；未寫入母表的要求，不得臨時加碼用來否決當前輪次。 撰寫原則：必須把正常、失敗、邊界、恢復、非法、重啟、不同步、證據、權限、責任、禁止事項全部寫清楚。 |

# 總則

本法典的目的，是把交易系統從『藍圖』提升為『可執行、可驗收、可追責、不可模糊解釋』的規則體系。

任何模型、代理、排程、人工審核官、或自動驗收腳本，只能依本法典的當前輪次規則判定 pass / fail。

未明文通過前，任何輪次不得被視為『差不多完成』、『大致可用』或『可以先往後做』。

# 第一編　全域硬規則

| 規則 | 法律效力 |
| --- | --- |
| 母表唯一性 | 所有檢查、排程、進度判定、opencode 指令，都必須嚴格依據 157 輪固定母表的當前輪既定標準。不得臨時猜測、補腦或提高標準。 |
| 禁止回頭加碼改判 | 若需提高標準，必須先正式修改母表，並於下一輪或 A 輪生效。不得用新標準否決已通過輪次。 |
| 嚴格順序執行 | 只允許處理當前輪次；前一輪未正式通過前，絕對禁止進入下一輪。 |
| 整輪重檢義務 | 每次審核都必須完整重檢當前輪，不得只看最後修改片段。任何遺漏即視為未通過。 |
| 永不放寬 | 關鍵驗收點只要有一項未完整達成，即判未通過。不存在『局部 OK』或『大致完成』。 |
| 進度一致 | 排程與手動檢查進度必須一致；本聊天室最新一次正式專案檢查輸出為唯一對外進度依據。 |
| 固定模板義務 | 所有專案檢查結果與 opencode 指令必須使用固定模板，且只能有單一 code block。 |
| 完成後義務 | 每輪完成後必須：自行檢查、 自行 commit、 自行 push。未完成三者之一，不得宣稱本輪完結。 |

# 第二編　證據規則

通過與否不能靠口頭陳述，必須有證據。證據效力由高到低如下：

| 證據等級 | 內容 |
| --- | --- |
| 一級證據（最強） | • 可重跑測試結果 • 固定驗收腳本輸出 • API response / state snapshot • Broker 對帳結果 • 不可變事件 log |
| 二級證據 | • commit diff • schema / contract 定義 • 自動回歸報告 • 對比報告 • 壓力測試摘要 |
| 三級證據 | • UI 實測截圖 • 明確頁面操作錄影 • 人工檢查清單 |
| 無效證據 | • 註解 • TODO • commit message 自稱完成 • AI 口頭保證 • 未驗證的單張截圖 |

# 第四編　所有輪次共同必須考慮的情境

| 情境類別 | 要求 |
| --- | --- |
| 正常情境 | 規格預期的主要成功路徑，必須逐條列出成功後哪些狀態改變、哪些狀態不得改變。 |
| 失敗情境 | 功能存在但結果錯、欄位存在但來源錯、UI 對但後端錯、主流程對但 recovery 錯等。 |
| 邊界情境 | 空資料、null、stale、timeout、部分成交、接近上限、接近收盤、首次啟動、最後一筆等。 |
| 恢復情境 | 異常解除、熔斷解除、recovery 升階、冷啟動授權、暫停後恢復。 |
| 重啟情境 | 系統重啟、restore、snapshot 還原、重啟後狀態是否重建、是否發生重送單。 |
| 非法情境 | 不該允許的模式切換、不該允許的設定修改、不該允許的權限、不該允許的資料回寫。 |
| 不同步情境 | 前端/後端/API/DB/引擎 各自看到不同 state、不同 mode、不同 position、不同 pnl。 |
| 證據情境 | 哪些輸出可作為驗收，哪些不可；驗收不完整時如何判 fail。 |

# 第五編　交易系統底層法典（高優先補強）

以下六大底層法典，建議提升為獨立輪次或獨立法典章節。若未獨立，後續多數高階功能都可能建立在不穩底盤上。

## A. 訂單生命週期狀態機法典

- 必須定義 created / pending_submit / submitted / acknowledged / partial_fill / filled / cancel_pending / cancelled / rejected / expired / reconcile_pending 等狀態。

- 必須明定所有合法與非法轉移，禁止未定義轉移默默放行。

- 必須處理部分成交、重複回報、回報順序顛倒、cancel 與 fill 競爭、重啟後重建。

## B. 現金、買力與交割資金帳本法典

- 必須有現金總帳、凍結資金、可用買力、未結算資金、已實現 / 未實現損益入帳規則。

- 委託送出、部分成交、撤單、拒單、交割日差異，都要明定資金如何變動。

## C. 冪等性與去重法典

- 所有高風險命令都應有唯一 key；retry 不可造成雙重送單；重播 fill event 不可重複入帳。

- restore / replay / recovery 之後，不可因流程重建而再執行舊指令。

## D. 不可變審計事件帳本法典

- decision / order / fill / cancel / reject / risk alert / mode change 都必須是追加式事件，不可覆寫歷史。

- replay、修正、對帳只能新增更正事件，不可回寫污染原始鏈。

## E. 市場日曆與交易時段法典

- 必須定義休市日、零股時段、盤前、盤中、盤後、特殊日、禁止下單時段與非交易時段允許任務。

## F. 競態、鎖與原子性法典

- risk 檢查與送單必須具原子性；position / order / cash 更新必須有鎖或序列化規則；多程序不可互相覆蓋。

# 第六編　模式、狀態與一致性法典

- 系統模式至少包含 observe / sim / paper / live / pause / recovery；任何新模式必須先寫入母表，否則視為非法。

- 模式切換必須前後端、API、引擎、持久化 state 同步完成；只切 UI 或只切後端都視為 fail。

- 前後端不得各自推導 mode；必須只有單一真實模式來源。

- recovery 不得只存在 transition 表；必須具備正式進入條件、正式離開條件、驗證方式與恢復期限制。

- 任何未定義模式切換，一律拒絕。

# 第七編　損益、成本與真實性法典

- 所有 realized pnl 必須來自唯一核心計算邏輯。

- 必須拆出 gross_pnl、fee、tax、spread/slippage/inefficiency、net_pnl。

- 模擬、回測、paper、shadow、live、強制平倉、部分平倉的口徑必須一致；若不一致，必須有明文例外法條。

- 前端不得自行再算另一套 pnl；顯示值必須由正式後端契約提供。

# 第八編　風控法典

## 通用原則

- 風控優先於策略；策略不能推翻硬上限。

- 當資料品質、執行品質、模式狀態、資金帳、健康檢查任一失真時，系統必須朝保守方向退。

## 至少必須存在的硬上限

- 單筆風險

- 單日風險

- 單週/單月風險

- 單股曝險

- 單策略曝險

- 組合總曝險

- 當沖未沖銷最壞交割能力

## no-trade 原則

- 事件風險未釐清不做

- regime 不清不做

- 資料 stale 不做

- execution 不可行不做

- 恢復期未升階前不做超出授權的交易

# 第九編　回測、模擬、shadow、live 隔離法典

- 歷史流、回放流、模擬流、live 流必須隔離；任何 replay/history 不得回寫 live state。

- 模擬成交器必須受市場制度規則約束；不得假設所有穿價都成交。

- shadow 與 live 差異必須持續記錄並可審計。

# 第十編　AI 治理與研究法典

- 六大 AI 只能提供理由、信心、分數與建議，不得繞過風控硬門直接下單。

- 任何 AI 權重、參數、風控規則、策略規則變更，都應進入 strategy CI/CD 驗證鏈。

- 研究結論不得直接升級至主系統；必須經 research gate、固定案例集、壓力測試、舊版/新版比較。

# 第十一編　輸出模板法典

所有正式專案輸出必須嚴格遵守固定模板，禁止改形、減欄、改用第二個 code block、或漏寫 REPLY_ID。

| 模板規則 | 內容 |
| --- | --- |
| 必須有 | • [PROJECT_CONTEXT_START] ... [PROJECT_CONTEXT_END] • [OPENCODE_START] • PART: 1/1 • 輪次判定 • 最新 commit • 未通過 / 通過 • 這次只修 • 通過標準 • 禁止 • 回報 • 完成後：自行檢查 / 自行 commit / 自行 push • REPLY_ID 置於 [OPENCODE_END] 前 |
| 明確禁止 | • 第二個 code block • 改用其他模板 • 用一般敘述取代正式輸出 • 缺少完成後三項 • 省略輪次判定或最新 commit |

# 第十二編　直接 Fail 條件總表

| Fail 類型 | 說明 |
| --- | --- |
| 跳輪 | 前一輪未正式通過卻進入下一輪。 |
| 雙真相來源 | 同一核心概念（mode、position、pnl、cash、order state）存在兩套正式來源。 |
| 局部修補冒充整輪完成 | 只修入口、不修主流程；只修 UI、不修後端；只修 happy path、不修非法情境。 |
| 沒有驗收證據 | 沒有固定測試、固定輸出、固定報告，卻宣稱通過。 |
| 用註解代替實作 | 以文件、註解、TODO、commit message 取代真正程式碼或契約變更。 |
| 未完成 commit / push | 未自行檢查、未 commit、未 push，仍宣稱本輪完成。 |
