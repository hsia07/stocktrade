NON-USAGE STALL STATE MACHINE (Alternative Monitoring Only)

## States
1. running
   - Entry: automode loop starts, auto_run=true, no PAUSE.flag
   - Exit: PAUSE.flag detected OR stall detected
   - Persistence: _tick_count increments, _metrics updated
   - Recovery: manual /start (no auto-resume)

2. stall_suspected
   - Entry: N consecutive ticks with NO phase transition AND NO last_action change AND NO new checkpoint
   - Exit: phase transition detected OR last_action changed OR checkpoint created OR manual /start
   - Threshold: N = 10 ticks (debounce: must persist for 10 consecutive ticks)
   - Persistence: stall_start_tick recorded, alert sent once
   - Recovery: manual /start only (no auto-resume)

3. auto_paused_by_stall_monitor
   - Entry: stall confirmed after debounce, PAUSE.flag created, pause_reason="auto_paused_by_stall"
   - Exit: manual /start clears PAUSE.flag and checkpoint
   - Persistence: checkpoint saved with stall info
   - Recovery: manual /start only (explicitly blocked auto-resume)
   - Telegram alert: sent on pause (with stall duration)

4. awaiting_manual_start
   - Entry: after auto_paused_by_stall, PAUSE.flag active
   - Exit: user issues /start command
   - Persistence: PAUSE.flag + checkpoint remain until /start
   - Recovery: ONLY manual /start (auto-resume BLOCKED)

## State Transition Rules
- running → stall_suspected: N ticks without phase transition (debounce: N consecutive)
- stall_suspected → running: phase transition detected (false positive resolved)
- stall_suspected → auto_paused_by_stall: stall confirmed after debounce period
- auto_paused_by_stall → awaiting_manual_start: PAUSE.flag set, alert sent
- awaiting_manual_start → running: ONLY via manual /start (auto-resume BLOCKED)

## Stall Threshold Recommendations
- tick_interval = 2.0s (from api_automode_loop.py:80)
- N = 10 ticks → 20 seconds without phase transition
- Debounce: must be sustained for N=10 consecutive ticks (prevents false positives)
- Consideration: construction phases may be longer → allow N=30 (60s) for construction

## Debounce/Multi-Signal Confirmation
- Require ALL 3 signals simultaneously:
  1. No phase transition (check _metrics['phase_transitions'])
  2. No last_action change (check state.runtime.json last_action timestamp)
  3. No new checkpoint (check checkpoint.json modification time)
- Sustained for N consecutive ticks (debounce)
- Only THEN transition to stall_suspected

## Auto-Pause Safety Boundary
- ONLY pauses at safe step boundary (like manual /pause)
- Saves checkpoint before pausing (checkpoint with stall info)
- Sets pause_reason="auto_paused_by_stall" (NOT usage-related)
- Does NOT modify trading core, server_v2.py, etc.
- Telegram alert includes: round_id, current_phase, stall_duration, pause_reason

## Telegram Alert Trigger
- Condition: stall confirmed after debounce (stall_suspected → auto_paused_by_stall)
- Content:
  - Round ID, current phase
  - Stall duration (tick count * tick_interval)
  - pause_reason: "auto_paused_by_stall"
  - Instruction: "Use /start to resume manually"
- Frequency: ONCE per stall event (not repeated)
- Caller: TelegramProgressNotifier.send_progress() or send_message()

## Why NO Auto-Resume
- usage_source_available = FALSE (no quota/reset info)
- Auto-resume would require: usage source with reset_time
- Policy: manual-only fallback (formal_status_code: usage_source_blocked)
- ONLY manual /start can clear auto_paused_by_stall
- This is a safety feature, not a limitation
