# 全自動修碼與驗收治理憲章 v1

## 第一條：目的
本憲章用於治理本地長時間自動修碼、自動驗收、自動 commit / push 流程，確保高自動化與高穩定性可同時成立。

## 第二條：核心原則
1. 執行體不得替自己宣判通過。
2. 寫 code 的 agent 不得決定 pass。
3. 產生模板的橋接器不得決定 pass。
4. 驗收腳本是第一裁判。
5. 高風險變更必須經過高階審查。
6. 未通過驗收不得 commit / push / 升輪。
7. 前一輪未過不得進下一輪。

## 第三條：角色
- ChatGPT：高階審核官 / 流程設計官
- opencode：格式化指令官 / 執行橋接官
- 本地 coding agent：執行官
- 驗收腳本：第一裁判
- 你：最終放行者

## 第四條：驗收類型
每輪至少包含：
1. 結構驗收
2. 行為驗收
3. 流程驗收
4. 回歸驗收

## 第五條：禁止事項
1. 寫 code 的 agent 不得將自身輸出視為通過證據。
2. 未有 manifest 的輪次不得執行。
3. 無報告、無證據、無 run record 的結果不得視為完成。
4. 未經核准不得碰紅區。
5. 未通過驗收不得自動 push 主分支。
6. 不得以註解 / 文件 / TODO 取代實作。
7. 不得以表面 UI 修補冒充核心修復。

## 第六條：高風險變更定義
符合以下任一者即為高風險：
- 交易核心
- 風控硬限制
- 狀態機與 mode transition
- broker adapter
- live credentials / secrets
- reconciliation / execution ledger
- 單一真實來源核心欄位
- 影響已通過輪次的共用基礎設施

## 第七條：升級人工
符合以下任一者必須升級人工：
- 同一類失敗連續重試超過上限
- 驗收腳本 pass 但高階審查判定疑似偷換概念
- 涉及紅區
- 涉及重大修法
- 涉及 live 權限
- 涉及資料一致性事故

## 第八條：合法等待
若當前輪因外部依賴阻塞，得進入合法等待，但不得宣稱通過，不得推進依賴該成果的後續輪次。
