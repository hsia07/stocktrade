## 三、整合後之基本執行原則

一、只要未明文通過，任何輪次不得被視為差不多完成、大致可用、可以先往後做，或得以後續輪次成果反推前輪已收口。

二、每次審核必須完整重檢當前輪，不得只看最後修改片段、單一摘要、單張截圖、單段 log 或 AI 口頭保證。

三、任何模型、代理、排程器、人工審核官、面板、wrapper、bridge、驗收腳本或自動化流程，只能依本文件與其明示吸收之正式法源判定 pass / fail / blocked / candidate_ready 等狀態。

四、凡屬高風險流程，必須保留可重跑測試、固定驗收腳本輸出、API response / state snapshot、不可變事件 log、commit diff、schema / contract 定義、回歸報告等正式證據。註解、TODO、AI 口頭保證、單張未驗證截圖不得作為有效通過依據。

五、任何輪次均應有明確禁區、前置依賴、必做實作清單、必要功能矩陣、必要證據矩陣與直接 FAIL 條件。缺一不可。

六、任何修法不得追溯否決既已正式通過之輪次；但自本文件生效後，所有涉及 04 所管治理補強範圍之新指令、新檢查、新排程與新自動化執行，均應依本文件為優先依據。

## 四、Phase 1–4 正式分期法典

第 1 條【Phase 1 定義】Phase 1 為第 1 輪至第 15 輪。其核心任務為先保命、先穩定、先一致，至少涵蓋穩定性、restore / validate、單一真實來源、時間一致性、健康檢查、熔斷 / 降級、狀態機、可觀測性、核心隔離、資料契約與 API schema 固定。

第 2 條【Phase 2 定義】Phase 2 為第 16 輪至第 45A 輪。其核心任務為讓交易鏈能跑通且可追責，至少涵蓋資料治理深化、市場制度、execution 模擬、真實淨損益、研究與回放、manual override、強風控與可審計成交真實度。

第 3 條【Phase 3 定義】Phase 3 為第 46 輪至第 128 輪。其核心任務為把交易與研究體系提升到成熟化、shadow / readiness / reconciliation、版本演進與長期運行準備階段。

第 4 條【Phase 4 定義】Phase 4 為第 129 輪至第 161 輪。其核心任務為多帳戶、營運化、告警中心、長期治理、交接與運營監控。

第 5 條【Phase 的法律效果】Phase 僅作分期治理與授權範圍之高階框架，不改變『單輪唯一主題、單輪證據、單輪追責、單輪 merge 控制』之基本原則。任何輪次是否可進 candidate、可進 review、可 merge，仍須依該輪完整法條判定。

第 6 條【Phase 授權與 panel 授權關係】若使用者明確授權某 Phase 的後續輪次自動運行，該授權僅表示可在該 Phase 內、於合法停止條件未命中前，依多輪候選預製模式自動把後續輪次各自做到 candidate；並不當然授權 approve / promote / merge / push 到 master。

## 五、多輪候選預製模式法典

第 7 條【制度目的】為兼顧高風險工程專案之治理嚴格性與長時自動化效率，允許在明確授權下，將多個後續輪次之成果先行自動收斂為各自獨立之 candidate，以供使用者回來後集中審核；但此制度不得改變正式通過、正式 merge、正式收口之人工最終放行原則。

第 8 條【定義】多輪候選預製模式，指代理、排程器、OpenCode、Aider 或其他輔助工具，於使用者明確授權之輪次範圍內，得依序完成各輪最小必要施工、驗證、證據收集與 candidate 建立，但不得因此視為該輪已正式通過。

第 9 條【啟動前提】本模式非預設常態流程。僅於使用者以可追溯方式，明確授權起始輪次、終止輪次、適用 Phase、禁止事項與停止條件時，始得啟動。

第 10 條【法律效果限制】本模式僅放寬『可先預製候選』之流程，不放寬正式通過、正式 merge、正式收口、正式進下一 Phase 等控制權。不得因後一輪 candidate 已完成，而推定前一輪已正式收口。

第 11 條【單輪獨立義務】在多輪候選預製模式下，每一輪仍必須保有獨立身份，不得混帳。至少應具備獨立 round_id、candidate_id、candidate branch 或等價隔離機制、evidence package、RETURN_TO_CHATGPT、formal status code、candidate commit 與可回滾審計鏈。

第 12 條【禁止冒充正式通過】任何代理、排程器、OpenCode、驗收腳本或人工審核官，不得因某輪已進 candidate、已完成測試、已建立 evidence package，便表述為該輪已正式通過、可直接 merge 或可無條件進下一輪。

第 13 條【可前推範圍】本模式下，後續輪得在前一輪尚未正式 merge 前，繼續進行下一輪之 candidate 預製；但後續輪次之法律狀態僅得為 technical_unfinished、blocked、environment_unavailable、candidate_ready_awaiting_manual_review 或其他已明文定義且不構成正式通過之狀態碼。

第 14 條【停止規則】只要任一輪出現 law_undefined、授權外檔案被觸及、hard gate 無法合法通過、任務尺寸超過可合理拆分範圍、審計鏈不足以保證可追責、或命中使用者明訂停止條件，後續自動預製應立即停止，不得自行恢復。

第 15 條【集中審核權】使用者得於多輪候選預製完成後，一次性審核多個輪次之 candidate；但集中審核僅屬審核便利，不改變各輪獨立性，不得因此將多輪視為單一輪次通過。

第 16 條【禁止整包自動 merge】除非法典另有更高層正式授權，否則多輪候選預製模式下，不得將多輪 candidate 視為可整包自動 merge 之集合。每輪仍應逐輪進行候選核准、review / promote、merge 決策、merge 執行與收口留痕。

第 17 條【與不可跳輪法之關係】本模式構成對『前一輪未正式通過前不得進入下一輪』之有限例外，但僅限於 candidate 預製層級，不及於正式通過、正式 merge、正式收口或下一 Phase 啟動。

## 六、面板 Start 與自動授權法典

第 18 條【正式啟動前提】只有在 panel Start / Resume 與實際後端腳本、bridge endpoint、wrapper、排程狀態寫入之間存在明確、可驗證、可重跑之綁定時，Start 才可被視為正式啟動與授權行為。若 Start 僅為前端 UI reset、文字變更或 drain timer 清理，不得視為正式授權。

第 19 條【Start 的法律效果】若正式啟動前提成立，且使用者於當前法典與 baseline 中已明示該 Phase 可啟用多輪候選預製模式，則使用者自面板按下 Start，應自動理解為：授權該 Phase 後續所有尚未處理輪次，在合法停止條件未命中前，可自動依序預製到各自 candidate。

第 20 條【Start 不構成之授權】面板 Start 不當然構成下列授權：approve、promote、merge、push master、越權碰 forbidden path、放寬 hard gate、放寬證據規則、跳過單輪 RETURN_TO_CHATGPT、或跳過任一輪完整法條重檢。

第 21 條【停止點】即使 Start 已授權該 Phase 後續輪次自動預製，只要任一輪未達 candidate_ready_awaiting_manual_review、命中停止條件、進入 review / merge 決策點、或需要新的權限判定，流程即須停止並等待使用者。

第 22 條【優先解釋】若對面板 Start 是否構成某項更大授權有疑義，一律採最保守解釋：僅視為 candidate 預製授權，不視為 merge、push 或下一 Phase 啟動授權。

## 七、候選門檻與完整功能判定法典

第 23 條【候選不是只看摘要】OpenCode 在判斷某輪是否可提交至 candidate 時，不得只看最後修改片段、commit message、單段摘要、單張 diff、單一測試或 AI 口頭回報；必須回到該輪完整法條，逐項檢查該輪要求的所有功能、所有必要契約、所有必要情境與所有直接 FAIL 條件。

第 24 條【缺一不可】凡屬該輪法條所明列之必要功能、必要 API / schema / state contract、必要成功情境、必要失敗情境、必要邊界情境、必要恢復情境、必要非法情境、必要證據與必要驗收腳本輸出，只要有一項未完成、未可運行、未驗證、未留痕，該輪即不得標記 candidate_ready。

第 25 條【功能完整性義務】候選門檻必須同時滿足：一、該輪唯一主題所有必做功能皆已具備；二、功能不僅存在，且可正常運行；三、必要契約與狀態一致；四、必要測試與驗證通過；五、evidence package 完整；六、hard gate 真正執行且 PASS；七、無 fabricated evidence；八、無越權修改。

第 26 條【整輪重檢義務】OpenCode 判定 candidate 時，必須以整輪法條為唯一依據，重新審核完整輪次，不得只看本輪最後一次施工的檔案集合。任何遺漏即視為未達候選門檻。

第 27 條【候選法定最低證據】每輪 candidate 至少應具備 task.txt、aider.log、candidate.diff、report.json、該輪可重跑測試結果、必要 state / API 證據、必要 event log、必要驗收文件與完整 RETURN_TO_CHATGPT 主體輸出。缺一不可。

第 28 條【形式與實質雙成立】就算 evidence package 齊全，若功能實質上未完整具備；或功能看似存在但必要證據不足；二者任一不足，皆不得進 candidate。

## 八、merge 前完整重審法典

第 29 條【merge 前不得只看候選摘要】當候選已進入 candidate_ready_awaiting_manual_review 或 review branch 階段，任何代理、OpenCode、ChatGPT 或高階審核者，在判斷是否可 merge 前，不得僅依 RETURN_TO_CHATGPT 摘要、candidate 名稱或單段 acceptance 摘要做決策。

第 30 條【merge 前完整重審義務】於判斷候選是否可以 merge 時，必須重新讀取 repo 內與該輪有關之實際檔案、測試、契約、驗收記錄、run record、evidence package 與該輪完整法條，重新審核一遍該輪是否完整具備所有要求功能且能正常運行。

第 31 條【重審範圍】merge 前重審至少應涵蓋：一、該輪唯一主題；二、所有必做功能；三、所有必要契約；四、所有必要情境與直接 FAIL 條件；五、實際 repo 檔案與測試；六、candidate / review branch 差異；七、hard gate 是否真實通過；八、是否仍存在未解決風險。

第 32 條【重審結論效力】若 merge 前重審發現該輪仍有功能缺漏、契約不一致、測試不穩定、證據不足、或任何直接 FAIL 條件，則即使該候選先前曾被標為 candidate_ready，也不得 merge。

第 33 條【高階審核官義務】ChatGPT 在給出 merge 建議或 merge 指令前，應明示其係基於完整重審，而非僅基於 OpenCode 摘要；若未完成完整重審，不得提供 merge 指令。

## 九、merge 前人工簽字義務法典

第 34 條【merge 前明示授權義務】當流程已進入 review branch，且客觀上已達到可由人工決定是否 merge 之決策點時，任何代理、排程器、OpenCode、ChatGPT 或其他輔助系統，不得預先提供 merge 指令，不得逕自推進 merge 流程，亦不得以預設同意、默示同意或流程慣例視為已獲授權。

第 35 條【簽字的法律效果】僅於使用者以可追溯方式明確表示『好』『同意』『簽字同意』或其他等價明示授權後，始得提供 merge 指令、執行 merge、或進入任何足以發生等價法律效果之後續流程。

第 36 條【不得以沉默替代】沉默、未反對、先前慣例、模糊語句、閱讀未回覆、或長時間未處理，均不得視為 merge 授權。

第 37 條【違反效果】若未先取得使用者明示簽字，即先行提供 merge 指令、建議直接合流、或執行 merge，視為越權推進正式收口；除非使用者事後追認，否則程序上應視為違規。

## 十、RETURN_TO_CHATGPT 正式主體輸出法典

第 38 條【正式主體輸出義務】RETURN_TO_CHATGPT 不是摘要區，而是正式主體輸出區。凡足以影響輪次判定、candidate 狀態、approve / promote / merge、blocked、technical_unfinished、candidate_ready、修法成立與否或下一步流程之資訊，均必須完整寫入 RETURN_TO_CHATGPT，不得僅存在於前文說明、人類可讀摘要或其他非正式區塊。

第 39 條【前文僅供輔助】正式輸出之前文 1–5 點、分段說明、表格、敘述性總結或其他人類可讀內容，僅得作為輔助閱讀材料，不得比 RETURN_TO_CHATGPT 更完整，不得承載正式主體結論，不得形成『前文較完整、正式區僅寫摘要』之輸出結構。

第 40 條【最低必備欄位】RETURN_TO_CHATGPT 至少應載明：round_id、task_type、reply_id、status 或 formal_status_code、candidate_id / branch / commit（若有）、files_modified 或 NONE、證據 / 驗證 / 測試結果、escalation_required、next_action，以及其他足以影響是否放行、是否退回、是否 blocked 之關鍵結論。

第 41 條【不一致之效果】若前文內容與 RETURN_TO_CHATGPT 不一致，或關鍵資訊僅存在於前文而未進入 RETURN_TO_CHATGPT，視為正式輸出無效，整份回報應退回重做，不得作為驗收、排程、候選核准、promotion、人工審核或輪次推進之依據。
