## 五、多輪候選預製模式法典

第 7 條【制度目的】為兼顧高風險工程專案之治理嚴格性與長時自動化效率，允許在明確授權下，將多個後續輪次之成果先行自動收斂為各自獨立之 candidate，以供使用者回來後集中審核；但此制度不得改變正式通過、正式 merge、正式收口之人工最終放行原則。

第 8 條【定義】多輪候選預製模式，指代理、排程器、OpenCode、Aider 或其他輔助工具，於使用者明確授權之輪次範圍內，得依序完成各輪最小必要施工、驗證、證據收集與 candidate 建立，但不得因此視為該輪已正式通過。

第 9 條【啟動前提】本模式非預設常態流程。僅於使用者以可追溯方式，明確授權起始輪次、終止輪次、適用 Phase、禁止事項與停止條件時，始得啟動。

第 10 條【法律效果限制】本模式僅放寬『可先預製候選』之流程，不放寬正式通過、正式 merge、正式收口、正式進下一 Phase 等控制權。不得因後一輪 candidate 已完成，而推定前一輪已正式收口。

第 11 條【單輪獨立義務】在多輪候選預製模式下，每一輪仍必須保有獨立身份，不得混帳。至少應具備獨立 round_id、candidate_id、candidate branch 或等價隔離機制、evidence package、RETURN_TO_CHATGPT、formal status code、candidate commit 與可回滾審計鏈。

第 12 條【禁止冒充正式通過】任何代理、排程器、OpenCode、驗收腳本或人工審核官，不得因某輪已進 candidate、已完成測試、已建立 evidence package，便表述為該輪已正式通過、可直接 merge 或可無條件進下一輪。

第 13 條【可前推範圍】本模式下，後續輪得在前一輪尚未正式 merge 前，繼續進行下一輪之 candidate 預製；但後續輪次之法律狀態僅得為 technical_unfinished、blocked、environment_unavailable、candidate_ready_awaiting_manual_review 或其他已明文定義且不構成正式通過之狀態碼。

第 14 條【停止規則】只要任一輪出現 law_undefined、授權外檔案被觸及、hard gate 無法合法通過、任務尺寸超過可合理拆分範圍、審計鏈不足以保證可追責、或命中使用者明訂停止條件，後續自動預製應立即停止，不得自行恢復。

第 15 條【集中審核權】使用者得於多輪候選預製完成後，一次性審核多個輪次之 candidate；但集中審核僅屬審核便利，不改變各輪獨立性，不得因此將多輪視為單一輪次通過。

第 16 條【禁止整包自動 merge】除非法典另有更高層正式授權，否則多輪候選預製模式下，不得將多輪 candidate 視為可整包自動 merge 之集合。每輪仍應逐輪進行候選核准、review / promote、merge 決策、merge 執行與收口留痕。

第 17 條【與不可跳輪法之關係】本模式構成對『前一輪未正式通過前不得進入下一輪』之有限例外，但僅限於 candidate 預製層級，不及於正式通過、正式 merge、正式收口或下一 Phase 啟動。

## 六、面板 Start 與自動授權法典

第 18 條【正式啟動前提】只有在 panel Start / Resume 與實際後端腳本、bridge endpoint、wrapper、排程狀態寫入之間存在明確、可驗證、可重跑之綁定時，Start 才可被視為正式啟動與授權行為。若 Start 僅為前端 UI reset、文字變更或 drain timer 清理，不得視為正式授權。

第 19 條【Start 的法律效果】若正式啟動前提成立，且使用者於當前法典與 baseline 中已明示該 Phase 可啟用多輪候選預製模式，則使用者自面板按下 Start，應自動理解為：授權該 Phase 後續所有尚未處理輪次，在合法停止條件未命中前，可自動依序預製到各自 candidate。

第 20 條【Start 不構成之授權】面板 Start 不當然構成下列授權：approve、promote、merge、push master、越權碰 forbidden path、放寬 hard gate、放寬證據規則、跳過單輪 RETURN_TO_CHATGPT、或跳過任一輪完整法條重檢。

第 21 條【停止點】即使 Start 已授權該 Phase 後續輪次自動預製，只要任一輪未達 candidate_ready_awaiting_manual_review、命中停止條件、進入 review / merge 決策點、或需要新的權限判定，流程即須停止並等待使用者。

第 22 條【優先解釋】若對面板 Start 是否構成某項更大授權有疑義，一律採最保守解釋：僅視為 candidate 預製授權，不視為 merge、push 或下一 Phase 啟動授權。

## 七、候選門檻與完整功能判定法典

第 23 條【候選不是只看摘要】OpenCode 在判斷某輪是否可提交至 candidate 時，不得只看最後修改片段、commit message、單段摘要、單張 diff、單一測試或 AI 口頭回報；必須回到該輪完整法條，逐項檢查該輪要求的所有功能、所有必要契約、所有必要情境與所有直接 FAIL 條件。

第 24 條【缺一不可】凡屬該輪法條所明列之必要功能、必要 API / schema / state contract、必要成功情境、必要失敗情境、必要邊界情境、必要恢復情境、必要非法情境、必要證據與必要驗收腳本輸出，只要有一項未完成、未可運行、未驗證、未留痕，該輪即不得標記 candidate_ready。

第 25 條【功能完整性義務】候選門檻必須同時滿足：一、該輪唯一主題所有必做功能皆已具備；二、功能不僅存在，且可正常運行；三、必要契約與狀態一致；四、必要測試與驗證通過；五、evidence package 完整；六、hard gate 真正執行且 PASS；七、無 fabricated evidence；八、無越權修改。

第 26 條【整輪重檢義務】OpenCode 判定 candidate 時，必須以整輪法條為唯一依據，重新審核完整輪次，不得只看本輪最後一次施工的檔案集合。任何遺漏即視為未達候選門檻。

第 27 條【候選法定最低證據】每輪 candidate 至少應具備 task.txt、aider.log、candidate.diff、report.json、該輪可重跑測試結果、必要 state / API 證據、必要 event log、必要驗收文件與完整 RETURN_TO_CHATGPT 主體輸出。缺一不可。

第 28 條【形式與實質雙成立】就算 evidence package 齊全，若功能實質上未完整具備；或功能看似存在但必要證據不足；二者任一不足，皆不得進 candidate。

## 八、merge 前完整重審法典

第 29 條【merge 前不得只看候選摘要】當候選已進入 candidate_ready_awaiting_manual_review 或 review branch 階段，任何代理、OpenCode、ChatGPT 或高階審核者，在判斷是否可 merge 前，不得僅依 RETURN_TO_CHATGPT 摘要、candidate 名稱或單段 acceptance 摘要做決策。

第 30 條【merge 前完整重審義務】於判斷候選是否可以 merge 時，必須重新讀取 repo 內與該輪有關之實際檔案、測試、契約、驗收記錄、run record、evidence package 與該輪完整法條，重新審核一遍該輪是否完整具備所有要求功能且能正常運行。

第 31 條【重審範圍】merge 前重審至少應涵蓋：一、該輪唯一主題；二、所有必做功能；三、所有必要契約；四、所有必要情境與直接 FAIL 條件；五、實際 repo 檔案與測試；六、candidate / review branch 差異；七、hard gate 是否真實通過；八、是否仍存在未解決風險。

第 32 條【重審結論效力】若 merge 前重審發現該輪仍有功能缺漏、契約不一致、測試不穩定、證據不足、或任何直接 FAIL 條件，則即使該候選先前曾被標為 candidate_ready，也不得 merge。

第 33 條【高階審核官義務】ChatGPT 在給出 merge 建議或 merge 指令前，應明示其係基於完整重審，而非僅基於 OpenCode 摘要；若未完成完整重審，不得提供 merge 指令。

## 九、merge 前人工簽字義務法典

第 34 條【merge 前明示授權義務】當流程已進入 review branch，且客觀上已達到可由人工決定是否 merge 之決策點時，任何代理、排程器、OpenCode、ChatGPT 或其他輔助系統，不得預先提供 merge 指令，不得逕自推進 merge 流程，亦不得以預設同意、默示同意或流程慣例視為已獲授權。

第 35 條【簽字的法律效果】僅於使用者以可追溯方式明確表示『好』『同意』『簽字同意』或其他等價明示授權後，始得提供 merge 指令、執行 merge、或進入任何足以發生等價法律效果之後續流程。

第 36 條【不得以沉默替代】沉默、未反對、先前慣例、模糊語句、閱讀未回覆、或長時間未處理，均不得視為 merge 授權。

第 37 條【違反效果】若未先取得使用者明示簽字，即先行提供 merge 指令、建議直接合流、或執行 merge，視為越權推進正式收口；除非使用者事後追認，否則程序上應視為違規。

## 十、RETURN_TO_CHATGPT 正式主體輸出法典

第 38 條【正式主體輸出義務】RETURN_TO_CHATGPT 不是摘要區，而是正式主體輸出區。凡足以影響輪次判定、candidate 狀態、approve / promote / merge、blocked、technical_unfinished、candidate_ready、修法成立與否或下一步流程之資訊，均必須完整寫入 RETURN_TO_CHATGPT，不得僅存在於前文說明、人類可讀摘要或其他非正式區塊。

第 39 條【前文僅供輔助】正式輸出之前文 1–5 點、分段說明、表格、敘述性總結或其他人類可讀內容，僅得作為輔助閱讀材料，不得比 RETURN_TO_CHATGPT 更完整，不得承載正式主體結論，不得形成『前文較完整、正式區僅寫摘要』之輸出結構。

第 40 條【最低必備欄位】RETURN_TO_CHATGPT 至少應載明：round_id、task_type、reply_id、status 或 formal_status_code、candidate_id / branch / commit（若有）、files_modified 或 NONE、證據 / 驗證 / 測試結果、escalation_required、next_action，以及其他足以影響是否放行、是否退回、是否 blocked 之關鍵結論。

第 41 條【不一致之效果】若前文內容與 RETURN_TO_CHATGPT 不一致，或關鍵資訊僅存在於前文而未進入 RETURN_TO_CHATGPT，視為正式輸出無效，整份回報應退回重做，不得作為驗收、排程、候選核准、promotion、人工審核或輪次推進之依據。

第 42 條【唯一正式依據】任何代理、排程器、人工審核官、驗收腳本或面板，於正式判讀時，應以完整且有效之 RETURN_TO_CHATGPT 為唯一正式主體輸出依據；前文僅供輔助閱讀，不得取代之。

第 43 條【回報交付協議——聊天室同步輸出義務】
一、當使用者明確要求「回報給 ChatGPT」或等價指令時，OpenCode 必須同步執行以下兩項交付：
　(1) 將完整回報內容寫入 latest_return_to_chatgpt.runtime.txt；
　(2) 將相同完整回報內容輸出至當前聊天室對話中，供使用者直接檢視。

二、僅在「自動模式」經使用者明確啟用時，OpenCode 得豁免聊天室輸出義務，僅執行檔案寫入。自動模式之啟用必須以可追溯方式記錄於 state.runtime.json 之 desired_action 或等價欄位。

三、禁止僅寫入檔案而不在聊天室呈現（自動模式除外）。違反本條視為回報交付不完整，該回報不得作為驗收、排程、候選核准、promotion、人工審核或輪次推進之依據。

四、本條為強制執行規範，不得因方便、效能、或「檔案已寫入即可」之慣例而省略聊天室同步輸出。

## 十一、OpenCode / Aider / 面板執行補充法典

第 43 條【OpenCode 角色】OpenCode 之正式角色為橋接官、任務格式官、回報官。其職責限於承接當輪唯一任務或已合法授權之多輪候選預製任務、在授權範圍內施工或派工、執行被允許之驗證、與輸出可審計結果。

第 44 條【OpenCode 禁止事項】OpenCode 不是自由施工員、不是環境修復員、不是最終放行者。除非法典明確授權，不得自行安裝套件、修改 PATH 或永久環境變數、清理 working tree、push、merge、宣布正式通過或擴權碰 forbidden path。

第 45 條【Aider 負荷上限】凡透過 Aider 施工之任務，原則上一輪只能有一個明確小任務，且應優先只授權一個檔案。若確有必要，最多只允許兩個高度相關之中小型檔案同輪送入。不得把超大單檔、核心交易主流程大檔、需要整 repo 理解之任務、或規劃 / 環境修復 / 改檔 / 驗證混成一次複合任務直接丟給 Aider。

第 46 條【Aider 最小上下文義務】每次呼叫 Aider 前，應優先以 --file、--message、--no-auto-commits、--no-dirty-commits 組成最小上下文；若仍超出負荷，應先拆任務而非強送。

第 47 條【證據守門】validate_evidence.ps1 或其後續正式替代腳本，屬 candidate 硬門之一部分。未真正執行且通過時，不得以人工摘要、手動比對或口頭保證取代。

第 48 條【Panel / wrapper 啟動鏈】所有桌面捷徑、bat/cmd/ps1 wrapper、bridge endpoint、panel 按鈕、-File 路徑與引號拼接，均屬正式啟動鏈的一部分。若存在空路徑、錯誤引號、腳本不存在或僅前端 reset，即屬啟動器故障，不得誤判為核心工具鏈可用。

## 十二、Phase 級主核心 Merge 治理法典

第 49 條【Phase 內單輪收口與 Phase 級收口之區別】Phase 內各輪得依正式流程進入 candidate、review、merge 到 work branch、push 到 origin/work branch；單輪進入 work branch，僅表示該輪在 Phase 工作主線內已完成收口，不等於該輪可直接進入 master / 主核心分支。

第 50 條【Phase 級總審核義務】Phase 內所有預定納入輪次皆完成單輪收口到 work branch 後，必須先執行 Phase 級總審核；未完成 Phase 級總審核前，任何輪次或整個 Phase 不得 merge 進 master / 主核心分支。

第 51 條【Phase 級主核心 Merge 之獨立性】Phase 級主核心 merge 屬獨立決策點，與單輪 merge 到 work branch 分屬不同授權層級。單輪已 merge 到 work branch，不得推定為該輪或該 Phase 可直接進入主核心。

第 52 條【禁止自動整包 Merge】OpenCode、排程器、代理或任何自動化流程，不得因 Phase 內多輪皆已進入 work branch，便自動或整包將該等輪次 merge 進 master / 主核心分支。

第 53 條【未完成輪次之阻塞效果】Phase 內若仍存在 candidate only、review only、technical_unfinished、blocked 或其他非正式收口狀態之輪次，原則上不得進行該 Phase 之主核心 merge；除非使用者明示授權例外處理並記錄理由。

第 54 條【使用者明示簽字義務】OpenCode 到達 Phase 級主核心 merge 決策點時，必須先停止流程，等待使用者以可追溯方式明示簽字同意。未獲使用者明示簽字前，OpenCode 不得提供主核心 merge 指令、不得執行 master merge、不得建議直接合流。

第 55 條【不得推定同意】不得因使用者曾簽字同意單輪 merge 到 work branch、或曾授權多輪候選預製、或長時間未處理，便推定使用者已同意 Phase 級主核心 merge。

第 56 條【work branch 定位】work branch（如 work/r006-governance）為 Phase 內收口工作主線，供多輪逐步匯集與 Phase 內整合測試之用；master / 主核心分支為正式發布與生產就緒之獨立保護層。

第 57 條【主核心 Merge 前檢查清單】判定是否可進行 Phase 級主核心 merge 前，必須確認：
一、該 Phase 預定納入輪次是否皆已收口到 work branch；
二、是否仍存在 candidate only、review only、technical_unfinished、blocked 狀態之輪次；
三、是否已完成 Phase 級總審核，確認各輪功能協同正常、無互相衝突、無回歸失效；
四、是否已取得使用者明示簽字同意。

第 58 條【違反效果】違反本章規則、未經 Phase 級總審核或使用者明示簽字即執行主核心 merge，視為越權推進正式收口，應立即停止並回報 blocked；已發生之 merge 應視為待覆審狀態，等待使用者追認或回滾。

## 十三、Shell / Git 非互動執行治理法典

第 59 條【非互動執行義務】OpenCode 執行 git、shell、merge、push、switch、verify 或其他命令列流程時，必須採非互動模式，不得打開 editor、pager、或等待任何人工輸入。

第 60 條【merge 命令非互動參數】凡執行 git merge，必須使用可避免互動之參數，例如 --no-edit、--no-ff、或等價非互動設定；不得以預設模式執行，導致可能進入互動編輯器等待人工輸入。

第 61 條【前置最小檢查義務】執行 switch、merge、push、verify 前，必須先做最小必要前置檢查：一、確認分支存在（git rev-parse --verify）；二、確認工作目錄狀態（git status --porcelain）；三、確認授權範圍內操作對象。未做前置檢查即執行高風險操作，視為程序瑕疵。

第 62 條【超時中止義務】shell 或 git 步驟若超過固定上限時間（建議 15 秒）仍未完成，必須立即停止並回報：一、technical_unfinished，或二、blocked。不得停在「思考中」、「處理中」或等價無限掛起狀態。

第 63 條【merge conflict 立即 abort】若 merge 遭遇 conflict，必須立即執行 git merge --abort，且不得自行進入交互式修復、不得嘗試自動解決衝突、不得詢問使用者如何處理。abort 後應回報 technical_unfinished，等待進一步指示。

第 64 條【失敗完整回報義務】若 shell 或 git 步驟失敗，必須完整回報：一、executed_command（實際執行之命令）；二、stdout（標準輸出）；三、stderr（錯誤輸出）；四、failure_point（失敗點描述）。不得只回報「失敗」或「出錯」而不提供具體證據。

第 65 條【禁止多輪猜測式嘗試】不得因前一次命令失敗，便自行進行多輪猜測式 shell 嘗試、變更參數重試、或切換替代方案。每次失敗應停止並回報，等待明確授權後方可進行下一步。

第 66 條【非互動規則之硬性】本章規則屬治理硬限制，適用於所有 OpenCode 執行之 git、shell、驗證、merge、push、switch、verify 任務。不得因方便、經驗、或「通常不會有問題」而省略非互動參數或超時保護。

## 十四、Governance Drift 治理與單輪分支管理規則

第 67 條【單輪分支禁改治理檔】單輪 candidate / review branch 原則上不得修改治理檔。所謂治理檔，至少包含：
一、CURRENT_GOVERNANCE_BASELINE.md
二、_governance/law/00_入口與使用規則/ 目錄下所有檔案
三、_governance/law/04_交易系統法典補強版_20260416_修正版.md
四、_governance/law/readable/ 目錄下所有檔案

第 68 條【治理檔修改屬性】治理檔修改屬治理任務，非單輪功能任務。單輪功能實作（如 R-007 silence detection、R-008 state machine、R-009 command priority）不得夾帶治理檔變更。

第 69 條【單輪分支開出基準】新的單輪 branch 必須從最新 work branch（如 work/r006-governance）開出；不得從落後於 work branch 的基底開出，以免帶入舊版治理文件。

第 70 條【Governance Drift 檢查義務】單輪進 approve / promote / merge 前，必須先做 governance drift 檢查。檢查方式：比較 target branch 與最新 work branch 之間的治理檔差異。

第 71 條【Governance Drift 判定標準】若 git diff --name-only <work_branch>...<target_branch> 命中第 67 條所列之治理檔集合，即判定 governance drift = true。

第 72 條【Drift 發現時之處理】發現 governance drift 時，必須 blocked，不得繼續 approve / promote / merge。OpenCode 必須立即停止並回報：
一、target branch 名稱
二、base work branch 名稱
三、drifted governance files 清單
四、建議下一步（rebase onto latest work / replay branch / split governance changes into separate task）

第 73 條【Rebase / Replay 義務】若 branch 基底落後 work branch，應先 rebase / replay / 重建，不得硬 merge。Rebase 後必須重新執行 governance drift check，直至 drift = false。

第 74 條【單輪 merge 內容限制】單輪 merge 優先只帶入該輪功能內容與該輪 evidence；治理檔不得被舊 branch 汙染回流。若 merge 後發現 governance file 被覆蓋為舊版本，視為 merge 失敗，應立即回滾或重建。

第 75 條【Governance Drift Check 之硬性】本章規則屬治理硬限制，適用於所有單輪 approve / promote / merge 流程。必須以可重跑之 hook 或驗證腳本落地，不得只憑人工記憶或口頭提醒。
