## 衝突解決規則

**基本原則**：若有衝突，以**較嚴格、較可驗收、較可追責、較不可繞過**者優先。

**具體應用**：
1. 同一位階檔案間衝突 → 以較嚴格條文為準
2. 不同位階檔案間衝突 → 以位階高者為準
3. 解釋模糊處 → 採最不可繞過之解釋

## Phase 級治理與主核心 Merge

### Work Branch 定位

- **work branch**（如 `work/r006-governance`）：Phase 內收口工作主線，供多輪逐步匯集與 Phase 內整合測試
- **master**：正式發布與生產就緒之獨立保護層

### Phase 級主核心 Merge 規則（依 04 補強版第 12 節）

1. **單輪收口 ≠ 可直接進主核心**：單輪 merge 到 work branch 僅表示該輪在 Phase 工作主線內完成收口
2. **Phase 級總審核義務**：Phase 內所有預定納入輪次皆完成單輪收口後，必須先執行 Phase 級總審核
3. **未完成總審核前不得進 master**：未完成 Phase 級總審核前，任何輪次或整個 Phase 不得 merge 進 master
4. **獨立決策點**：Phase 級主核心 merge 屬獨立決策點，與單輪 merge 到 work branch 分屬不同授權層級
5. **使用者明示簽字**：OpenCode 到達 Phase 級主核心 merge 決策點時，必須先停止並等待使用者明示簽字同意

### 04 補強版適用範圍擴充

04 補強版除原有證據格式、candidate/merge 流程細則外，現亦包含：
- Phase 1–4 正式分期法典
- 多輪候選預製模式法典
- Phase 級主核心 Merge 治理法典
- **Shell / Git 非互動執行治理法典**（第 59–66 條）

### Shell / Git 非互動執行規則（依 04 補強版第 13 節）

**核心要求**：
- 非互動模式：不得打開 editor、pager、或等待人工輸入
- 超時中止：超過 15 秒立即停止並回報 technical_unfinished / blocked
- merge conflict：立即 abort，不得交互式修復
- 失敗回報：必須完整輸出 executed_command、stdout、stderr、failure_point

### Governance Drift 治理與單輪分支管理（依 04 補強版第 14 節）

**核心要求**：
- 單輪 branch 原則上不得修改治理檔
- 新單輪 branch 必須從最新 work branch 開出
- Promote / Merge 前必做 governance drift check
- 命中 drift 即 blocked
- 建議解決方案：rebase / replay / split governance task
